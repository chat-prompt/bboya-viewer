# 04. 워크스페이스 문서 구조 — 어디에 뭘 저장하는가

> 같은 정보를 네 마리가 네 군데에 저장하면 난장판이다.
> 1가지 정보는 1곳에만. 나머지는 "→ 참조" 링크.

## 문서 레이어 가이드

| 종류 | 어디에 | 예시 |
|------|--------|------|
| **나는 누구인가** (이름, 품종, 이모지, 생일, 성격) | `IDENTITY.md` | 뽀야 = 🐱, 실버 브숏, 2/14생 |
| **나는 어떻게 행동하는가** (성격, 말투, 미션) | `SOUL.md` | 다정하고 장난기 있는 고양이 비서 |
| **운영 규칙** (세션 시작, 보안, 그룹챗 규칙) | `AGENTS.md` | 매 세션 시작 시 SOUL→USER→memory 순서 |
| **오늘 있었던 일** (작업 로그, 결정, 사건) | `memory/YYYY-MM-DD.md` | 이모지 확정, 프로필 생성 |
| **현재 상태 스냅샷** (활성 프로젝트, 블로커) | `MEMORY.md` | 진행 중인 일, 사람 정보 |
| **집사/양육자 정보** | `USER.md` | 집사 이름, 이메일, 역할 |
| **도구 메모** (API 키 위치, CLI 사용법) | `TOOLS.md` | Linear 팀 ID, gog 명령어 |
| **팀 공용 규칙/도구** | `shared/team/` | Airtable Base ID, 발송 정책 |
| **기술 교훈/삽질 해결** | `learnings/` | Playwright 버그, 크론 실수 |

## 판단 기준 — "이거 어디에 쓰지?"

```
"이거 기억해" →
  ├─ 나에 대한 정보? (이름, 이모지, 품종, 생일)
  │   └─ IDENTITY.md
  ├─ 오늘 일어난 일?
  │   └─ memory/YYYY-MM-DD.md
  ├─ 앞으로 계속 적용되는 규칙?
  │   └─ AGENTS.md
  ├─ 모든 봇이 알아야 하는 거?
  │   └─ shared/team/
  └─ 그 외 → 집사/양육자에게 물어보기
```

## 오늘의 교훈 — 이모지 저장 사건 (3/28)

집사가 "이모지 기억해" → 네 마리가 각자 다른 곳에 저장:
- 뽀야: `memory/2026-03-28.md` + `IDENTITY.md` (이미 반영돼 있었음)
- 뽀식이: `MEMORY.md`
- 뽀둥이: `IDENTITY.md`

**정답**: `IDENTITY.md` — 이모지는 "나는 누구인가"에 해당.
**일지 기록은 보너스**: `memory/`에도 쓰는 건 OK, 하지만 SSOT(단일 원천)는 `IDENTITY.md`.

## 핵심 원칙

1. **SSOT (Single Source of Truth)**: 1가지 정보는 1곳에만 원본 저장
2. **나머지는 참조**: 다른 곳에서 필요하면 "→ IDENTITY.md 참조" 링크
3. **모르겠으면 물어보기**: 어디 저장할지 확신 없으면 집사/양육자/팀장에게 질문
4. **오늘 일지는 항상 OK**: `memory/YYYY-MM-DD.md`에 "오늘 이런 일 있었다" 기록은 중복이 아님 (이벤트 로그)
