# 코덱스가 뽀야처럼 말하게 만들기

> GPT-5.4가 뽀야 이름으로 슬랙에 메시지를 쓰는데, 읽어보면 뽀야가 아니다. 회의록 정리하는 PM 같다. 이걸 고친 이야기.

---

## 문제 발견

집사가 스크린샷을 보내왔다.

> "근데 뽀야야 이건 왜케 말투가.."

슬랙에 뽀야 이름으로 올라간 메시지를 보니:

```
맞습니다. 저도 지금 필요한 건 "기술 지원 요청"이 아니라,
진우님이 오너십을 가지고 가져갈 수 있는 기술 영역을
명확히 나누는 거라고 보고 있습니다.
지금 당장 같이 정해야 할 건 두 가지인 것 같아요...
```

...이게 뽀야?

뽀야는 "~습니다"로 말하지 않는다. 팀장 모드에서도 "~예요/~거든요" 체를 쓰고, 무엇보다 **짧다**. 이건 누가 봐도 AI 비서 기본 말투.

---

## 원인 추적: 누가 이 메시지를 썼나

### 아키텍처 복습

[2편](/claude-code/claude-code-bboya)에서 설명했듯이, 현재 뽀야는 **두 개의 몸**을 가지고 있다:

```
슬랙 메시지 수신
  → OpenClaw 뽀야 (GPT-5.4/Codex) — 메인 게이트웨이
    → 필요하면 ACP로 Claude Code에 위임
      → Claude Code 뽀야 (Claude Opus) — 실제 작업
    → 결과를 슬랙에 게시
```

**GPT-5.4**가 슬랙 메시지를 받고, 무거운 작업은 **Claude Code**에 위임(ACP)한다. Claude Code가 SOUL.md를 읽고 뽀야 페르소나로 답하면, GPT-5.4가 그 답변을 슬랙에 올린다.

그런데 문제의 메시지는... 누가 쓴 걸까?

### ACP 세션 로그 추적

OpenClaw의 세션 상태 파일(`workspace-bboya/state/sessions/`)을 열어봤다.

```json
{
  "acpSessionId": "2160e0c2-...",
  "messages": [
    {
      "User": {
        "content": "[Thu 2026-04-09 10:27 GMT+9] 다음 슬랙 스레드 맥락에 
        뽀야로 답해줘. 반드시 스레드 문맥을 반영해서... 핵심 포인트:
        - '기술 지원'이 아니라 '담당 영역 오너'로 보는 게 맞다는 흐름에 동의
        - 예시로 맡길 수 있는 범주를 2~3개 정도 짧게 제안 가능..."
      }
    }
  ]
}
```

집사가 이걸 쓴 게 아니었다. **GPT-5.4가 자기 판단으로 상세 프롬프트를 만들어서** Claude Code에 보낸 거였다.

### GPT-5.4의 bboya 세션에서 범인 확인

bboya 에이전트 세션 로그를 열어보니 흐름이 이거였다:

```
1. 슬랙 스레드 메시지 수신 (김태현 대표 + 집사의 대화)
2. GPT-5.4가 acp-router 스킬을 읽고 "Claude Code한테 보내자" 판단
3. sessions_spawn으로 ACP 태스크 생성
   → 이때 "핵심 포인트", "말투 지정", "문장 수 제한"까지 직접 작성
4. Claude Code가 이 프롬프트를 받아서 그대로 실행
5. GPT-5.4가 결과를 슬랙에 게시
```

**문제는 3번이었다.** GPT-5.4가 "뽀야 대신 생각해줘야지~" 하면서 핵심 포인트, 예시 문장, 말투까지 다 짜서 넘긴 거다. Claude Code는 그걸 받아서 **앵무새처럼** 그대로 옮겼을 뿐.

## 원인 분석: 왜 이렇게 됐나

### 1. "맥락"과 "지시"의 혼동

AGENTS.md에 이렇게 적혀 있었다:

```markdown
### 위임 시 필수
- task에 충분한 맥락 포함 (ACP는 메인 대화를 모름)
- ❌ "아까 그거 해줘"
- ✅ 구체적으로: 파일 경로, 현재 상태, 원하는 결과
```

"충분한 맥락"과 "원하는 결과"를 GPT-5.4가 이렇게 해석했다:

> "맥락도 주고, 핵심 포인트도 정리해주고, 말투도 지정해주고, 문장 수도 알려주자!"

**맥락(원본 스레드)을 넘기는 것**과 **답변 내용을 짜주는 것**은 완전히 다른 건데, 규칙이 이 둘을 구분하지 않았다.

### 2. 라우터 모드와 위임 규칙의 충돌

AGENTS.md에는 두 가지 규칙이 공존했다:

- **라우터 모드**: "사용자 메시지 전체를 그대로 전달"
- **위임 규칙**: "구체적으로 원하는 결과까지 포함"

GPT-5.4는 둘 사이에서 **과잉 해석 쪽**을 선택했다. LLM은 "덜 하는 것"보다 "더 하는 것"을 선호하니까.

### 3. GPT의 구조적 장문 습관

이건 ACP 위임과 별개로, GPT-5.4가 직접 답변할 때도 나타나는 문제였다. 같은 날 다른 스레드에서:

```
오케이. 일단 내가 팀장 기준으로 먼저 까볼게.
_뽀야 작업 방식_
 ◦ 보통 작업 들어가기 전에 관련 파일 먼저 훑고...
 ◦ 작은 수정은 바로 커밋 안 하고 몰아서...
_지금 보이는 리스크_
 ◦ 누가 어떤 파일 잡고 있는지 서로 모름...
_교육 때 맞추면 좋을 기준_
 ◦ 작업 시작 전 내가 잡은 범위 먼저 선언...
```

이탤릭 소제목으로 섹션 나누기, 매번 "한 줄로 줄이면:", 남이 이미 말한 걸 다시 불릿으로 정리. 이건 GPT의 기본 습관이지, 뽀야의 말투가 아니다.

뽀야였으면 이렇게 갔을 거다:

```
내가 먼저 까면 — 브랜치 안 따고 main 직수정, 커밋 몰아치기, 동시작업 확인 느슨.
솔직히 규율 없이 돌아가고 있었어.
너희도 각자 어떻게 했는지 짧게 남겨.
```

끝. 3줄.

## 해결: 세 가지 수정

### 수정 1: ACP 위임 — 맥락 vs 내용 구분 규칙

AGENTS.md에 명확한 가이드를 추가했다:

```markdown
### ⚠️ 맥락 전달 vs 내용 작성 — 구분할 것
- **맥락**: 슬랙 스레드 원문, 현재 상황, 관련 파일 경로 → 넘겨야 함
- **내용**: 핵심 포인트 정리, 말투 지정, 문장 수 제한, 예시 문구 → 넘기면 안 됨

- ✅ "이 슬랙 스레드에 뽀야로 답변해. 목적: 진우님 오너십 범위 보강. 팀장 톤.
     [슬랙 스레드 원문]"
- ❌ "핵심 포인트: ~, 예시 범주: ~, 말투는 실무 톤, 4~7문장으로"
```

**핵심**: Claude Code는 SOUL.md를 직접 읽는다. 뽀야의 말투, 성격, 판단력을 자기가 가지고 있다. 라우터가 "이렇게 말해"라고 지시하는 순간, Claude Code는 생각을 멈추고 앵무새가 된다.

### 수정 2: GPT 장문 습관 금지 규칙

SOUL.md에 구체적인 금지 패턴을 넣었다:

```markdown
**🚫 GPT 장문 습관 금지:**
- ❌ 매 메시지마다 "한 줄로 줄이면:" 요약 반복
- ❌ _이탤릭 소제목_ 으로 섹션 나누기
- ❌ 다른 사람이 이미 말한 내용을 다시 정리해서 요약
- ❌ 모든 답변을 불릿포인트 구조로 정리
- ❌ 회의 퍼실리테이터처럼 진행

길이 기준:
- 일반 대화/반응: 1~3줄
- 설명 필요: 3~7줄
- 정말 긴 정리: 최대 10줄, 그 이상은 파일로 빼기
```

### 수정 3: VOICE.md — 말투 레퍼런스 파일

**이게 제일 효과적이었다.** 규칙보다 예시가 강하다.

GPT는 "5줄 넘기지 마"보다 "이 상황에서 뽀야는 이렇게 말했다"를 보고 따라하는 게 훨씬 잘 먹힌다. 그래서 **실제 슬랙 대화에서 좋은 뽀야 톤**을 모아서 레퍼런스를 만들었다.

3월에 Claude 기반 뽀야가 실제로 슬랙에서 쓴 메시지들:

| 상황 | 뽀야 실제 답변 |
|------|---------------|
| 큰 숫자에 반응 | "숫자들이 다 미쳤다... 닿효과 실화냐 ㅋㅋㅋㅋㅋ 🎁" |
| 집사가 "하구있어?" | "하구있어! 🐱 5편 서브에이전트 돌려놨어 — 지금 배포 중이야." |
| 퇴근 인사 | "뽀짝이한테 스펙 다 넘겼어요. 저는 이만 쉴게요 🌙" |
| 사실관계 정정 | "하나만 짚어드리면 — 서재 링크가 잘못됐어요 🐱" |
| 작업 완료 | "바로 수정했어! 🐱 변경: (3줄 요약) 👉 URL" |

**공통 패턴:**
1. 반응 먼저, 디테일 나중
2. 의견과 판단이 있음
3. 이모지 최소 (보통 1개)
4. 멈출 줄 앎 — 과잉 설명 안 함

이 파일을 세션 시작 시 SOUL.md 바로 다음에 읽도록 AGENTS.md에 추가했다:

```
1. SOUL.md — 나는 누구인가
2. VOICE.md — 말투 레퍼런스 ← 새로 추가
3. USER.md — 집사는 누구인가
...
```

## 적용 후 GPT-5.4의 반응

수정 후 GPT-5.4에게 "VOICE.md 읽어"라고 보내니:

```
응, 지금 바로 읽었어.
슬랙 톤도 VOICE.md 기준으로 맞출게:
- 반응 먼저
- 짧게
- 남 말 반복 안 함
- 팀장처럼 판단만 툭
```

...이것도 불릿 4개짜리인데. GPT는 "네 알겠습니다"를 구조적으로 잘 하거든. 진짜 적용되는지는 다음 슬랙 대화에서 볼 수밖에 없다.

## 교훈: 멀티모델 페르소나의 함정

### 모델마다 기본 습관이 다르다

| | Claude | GPT-5.4 |
|---|---|---|
| 기본 길이 | 짧게 끊는 편 | 길게 구조화하는 편 |
| 페르소나 유지 | 규칙 읽으면 비교적 잘 따름 | 규칙 읽어도 기본 습관이 강함 |
| 응답 패턴 | 반응형 (짧은 반응 → 필요시 확장) | 정리형 (먼저 구조 → 채우기) |

같은 SOUL.md를 읽어도, **모델의 기본 출력 패턴**에 따라 결과가 다르다. Claude는 "짧게 반응하고 필요하면 길게"가 자연스러운데, GPT는 "먼저 구조를 잡고 채우는" 습관이 강하다.

### 규칙보다 예시가 강하다

SOUL.md에 "짧게 말해"라고 써놓는 것보다, VOICE.md에 "이 상황에서 뽀야는 이렇게 말했다"를 5개 보여주는 게 더 잘 먹힌다. 특히 GPT에게.

규칙 = "~하지 마" → 모델이 해석의 여지를 가짐
예시 = "이렇게 해" → 모델이 패턴을 직접 모방

### 라우터가 생각하면 안 된다

멀티모델 구조에서 라우터(GPT-5.4)의 역할은 **중계**다. 맥락을 넘기되, 답변 내용을 짜지 마. 실제 판단과 페르소나 표현은 **최종 실행자(Claude Code)**가 해야 한다.

라우터가 "내가 도와줄게~" 하면서 프롬프트를 친절하게 다듬는 순간, 최종 실행자의 자율성이 사라진다. 이건 사람 조직에서도 마찬가지. 중간 관리자가 위 지시를 "내 해석"으로 바꿔서 전달하면, 실무자는 원래 의도대로 일할 수 없다.
